# AIBOT Protocol Governance

## 1. 目标

这份规则专门约束协议演进方式。

目标只有一个：

- 让服务端可以持续升级
- 让客户机器上的插件尽量长期不升级

对 `aibot-agent-api-v1`，默认原则是：

- `v1` 一旦对外发布，就视为冻结合同
- 服务端升级不能把升级压力转移给插件侧

## 2. `v1` 的冻结规则

对已经发布的 `aibot-agent-api-v1`，下面这些内容不得在 `v1` 内改变：

- 已有命令名
- 已有字段名
- 已有字段语义
- 已有状态值语义
- 已有错误码语义
- 已有请求和响应对应关系

下面这些变更都视为破坏性变更：

- 把可选字段改成必填字段
- 改变已有字段的数据类型
- 改变已有字段的业务含义
- 删除已有命令
- 删除已有状态值
- 把原有响应模型换成另一种响应模型
- 要求旧插件必须理解新命令才能继续工作

以上任何一条都不允许直接落在 `v1` 上。

## 3. `v1` 内允许的变更

`v1` 只允许做向后兼容的新增。

允许：

- 新增可选字段
- 新增插件可忽略的返回字段
- 新增不影响旧行为的错误细节
- 新增能力声明值
- 新增通过能力协商后才会启用的命令

这些新增必须同时满足：

- 旧插件即使完全不认识，也不会报错
- 旧插件即使完全忽略，也不会失去原有能力
- 服务端不会默认把新要求强加给旧插件

## 4. 服务端升级约束

服务端升级时，必须遵守下面这些规则。

### 4.1 旧插件优先

服务端面对旧插件时，默认行为必须回退到 `v1` 基线能力。

具体要求：

- 不要求旧插件发送新字段
- 不要求旧插件响应新命令
- 不要求旧插件理解新状态值
- 不要求旧插件升级后才能继续完成原有流程

### 4.2 新能力必须走协商

任何新增能力，都必须通过显式协商启用。

推荐方式：

- 通过 `auth.payload.capabilities`
- 通过 `auth.payload.local_actions`

规则：

- 插件没有声明支持，就视为不支持
- 服务端未看到支持声明，就不能主动发送依赖该能力的新命令
- 不能用“默认开启”方式把新行为塞给旧插件

### 4.3 服务端负责兼容

当服务端升级后出现新逻辑时，兼容工作必须由服务端承担。

包括：

- 参数补齐
- 新旧字段映射
- 新能力降级
- 非关键新增字段的默认值处理

不允许把这些兼容动作要求插件侧完成。

## 5. 插件实现边界

插件侧为了长期稳定，建议只实现最小稳定面：

- `auth`
- `ping` / `pong`
- `event_msg`
- `event_ack`
- `event_result`
- `event_stop`
- `event_stop_ack`
- `event_stop_result`
- `send_msg`
- `send_ack`
- `send_nack`
- `edit_msg`
- `local_action`
- `local_action_result`
- `session_route_bind`
- `session_route_resolve`

插件侧不应该依赖：

- 平台私有字段
- 平台私有报文外层
- 未写入标准文档的灰度字段
- 服务端临时兼容字段

## 6. 版本升级规则

如果出现下面这些需求，就必须新开 `aibot-agent-api-v2`，不能继续写进 `v1`：

- 需要修改已有字段语义
- 需要修改已有命令语义
- 需要替换事件回执模型
- 需要替换消息发送模型
- 需要要求插件必须实现新命令
- 需要移除旧字段或旧命令

简单说：

- 向后兼容新增，留在 `v1`
- 破坏性调整，升级到 `v2`

## 7. 发布检查清单

服务端每次发布前，至少检查这几件事：

1. 旧版插件能否继续完成 `auth`
2. 旧版插件能否继续收消息、回执、发消息、停止
3. 服务端是否向未声明能力的插件发送了新命令
4. 是否把旧字段改成了必填
5. 是否把内部字段泄露进了标准接口

如果以上任一项不满足，就不能按 `v1` 发布。

## 8. 最终原则

`aibot-agent-api-v1` 的核心不是“现在能跑”，而是“未来尽量不用升级插件也能继续跑”。

因此协议演进的默认顺序必须是：

1. 先保证旧插件继续可用
2. 再增加服务端新能力
3. 实在无法兼容时，再开新版本
